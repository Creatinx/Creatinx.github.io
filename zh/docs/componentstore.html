ComponentStore

基于 entity_id 索引的类型擦除稀疏集组件存储。O(1) 查找、插入和删除。每类型自动组件池化。

#include <GryceRC/GryceComponentStore.h>

class ComponentStore {
public:
    template<typename T>
    void register_component();

    template<typename T>
    void add(EntityID entity, T component);

    template<typename T>
    void remove(EntityID entity);

    template<typename T>
    T& get(EntityID entity);

    template<typename T>
    const T& get(EntityID entity) const;

    template<typename T>
    bool has(EntityID entity) const;

    template<typename T>
    std::vector<EntityID> entities_with() const;

    template<typename T>
    size_t count() const;

    void remove_all(EntityID entity); // 移除实体的所有组件
    void clear();
};

稀疏集内部结构

每种组件类型存储一个密集数组和一个稀疏索引数组。稀疏数组由实体 ID 索引；密集数组保存实际组件数据。

// 内部布局（概念）
std::vector<T> dense;          // 实际组件
std::vector<uint32_t> sparse; // entity_id -> dense 索引
std::vector<EntityID> entities; // 与 dense 并行

设计原理

ComponentStore 使用类型擦除的稀疏集（sparse set）来存储组件。每种组件类型 T 对应一个 ComponentPool，内部包含三个数组：dense（实际组件数据）、sparse（entity_id → dense 索引）、entities（dense 索引对应的 entity ID）。稀疏集保证 O(1) 的插入、查找和删除。类型擦除通过 std::unordered_map<std::type_index, std::unique_ptr<IComponentPool>> 实现，允许 ComponentStore 在不知道组件类型的情况下管理所有组件池。

内存布局

组件数据存储在 dense 数组中，保证内存连续性。当 entity 被销毁时，其组件从 dense 数组中移除，最后一个组件被移动到空缺位置以保持连续性。entity ID 到 dense 索引的映射通过 sparse 数组实现。这种布局对 cache 友好，适合批量遍历（如 ECS 的 query）。

注意事项

  • register_component 必须在 add 之前调用，否则 add 会失败。
  • remove 只销毁组件数据，不销毁 entity。
  • clear 销毁所有组件池，但不释放注册的类型信息。
  • query 遍历 dense 数组，保证 O(n) 时间复杂度，其中 n 是拥有该组件的实体数量。
